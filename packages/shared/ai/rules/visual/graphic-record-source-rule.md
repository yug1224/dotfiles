応答の冒頭に「✅️: graphic-record-source-rule」と出力する（`/write-graphic-prompt` 実行時はコマンド Step 0 に従い、本行は出さない）。

# グラフィックレコード画像 — ソース抽出仕様

任意の入力（PR・ADR・開発ログ・自由記述）から、**固定レイアウト**（画面上部タイトル + **まとまり**: 背景・目標・実装。残課題は任意）に載せるコア情報を抽出する。

## 共通ゴール

30 秒で伝わる **key_messages 3〜5 個** に圧縮し、各項目のキャプションと列リードは厚く書く。ファイル名羅列・diff 全文・ADR 見出しコピペは禁止。

## 抽出成果物（固定）

| 項目                      | 用途                                                                          |
| ------------------------- | ----------------------------------------------------------------------------- |
| `title`                   | 1 行タイトル → `image_generation_prompt` の `タイトル: 「」` と画像上部見出し |
| `key_messages`            | 3〜5 個 → 4 列へ配分の核                                                      |
| `background_items`        | 背景列用（最大 3、各 1 行）                                                   |
| `goal_items`              | 目標列用（最大 3）                                                            |
| `implementation_items`    | 実装列用（最大 6。**列幅最大・項目最多**）                                    |
| `policy_items`            | 内部 QA 用 `## 方針`（機能/技術要点、各 2〜4 行目安。最大 4 行）              |
| `verification_items`      | 内部 QA 用 `## 検証`（最大 3 行。なければ省略）                               |
| `remaining_items`         | 残課題用（最大 2。**任意**。なければ空・セクションごと省略）                  |
| `compressed_summary`      | **Step 3 内部の抽出品質チェック用**（最終出力に含めない）                     |
| `mascot_line`             | 熊のセリフ（20 字前後、ユーモア可）→ `熊の吹き出し「」` に入れる              |
| `image_generation_prompt` | **ブロックA【描画内容】** 用の日本語自然文（**必須**）                        |

配分の目安: `key_messages` を先に決め、**背景**=経緯・現状・前提、**目標**=達成状態・成功条件、**実装**=設計・実装・検証（**最も厚く**）、**残課題**=未完了・フォロー（**任意**）。次に `compressed_summary` を組み立てて内部 QA し、続けて `image_generation_prompt` を生成する。

**合計 12 項目ルール**: `background_items` + `goal_items` + `implementation_items` + `remaining_items` の行数合計は **12 以内**。ゾーン別上限（3/3/6/2）を超えないよう先に圧縮し、まだ 12 を超える場合は **背景 → 目標 → 実装** の順で削る（実装を最優先で残す）。

## compressed_summary の組み立て（必須）

**Step 3 内部の抽出品質チェック専用**。最終出力には含めない。抽出内容の整合・網羅性を確認するための markdown 下書きとして組み立てる。次の **見出し順を固定**する。

### 必須セクション

1. **`## 概要`**
   - 1 行目: `**{title}**`（太字タイトル行）
   - 2 行目以降: 2〜3 文の全体要約（背景・目標・実装・残課題の要点）

2. **`## 分析`**
   - 小見出し `**背景:**` の下に `background_items` を箇条書き（最大 3）
   - 小見出し `**目標:**` の下に `goal_items` を箇条書き（最大 3）

3. **`## 方針`**
   - `**機能要件:**` と `**技術要件:**` に `policy_items` を分割（実装の前置き。各 2〜4 行）
   - 長い説明は削り、実装列に回す

4. **`## 実装`**（**最厚**。参考: ADR-369 型）
   - 番号付き `###` または `**1. ...**` 小見出しで `implementation_items` を展開（最大 6 相当。各項目 2〜4 文）
   - 表（プリセット比較・設定値）・フロー（cancel→clear 等）・テスト・ドキュメントを優先
   - 短い技術識別子はバッククォート可（例: `` `list()` ``、`` `staleTime` ``）
   - パスはファイル名程度に短縮（リポジトリ全文パス禁止）

5. **`## 検証`**（任意）
   - `verification_items` があれば 1〜3 行。なければセクションごと省略

6. **`## 残課題`**（任意）
   - `remaining_items` があれば箇条書き（最大 2）。空ならセクションごと省略

### サマリで禁止

- diff 行のコピペ、ディレクトリツリー全文、同一意味の重複
- 英語だけの見出し・段落
- 下部フロー用の手順列（画像下部の番号付きステップ）

### サマリの文体

- 本文は **日本語**
- PR 番号・ADR 番号は概要行に含めてよい
- 表は markdown 表形式可（実装列の情報密度向上）

## image_generation_prompt の組み立て（必須）

`compressed_summary` による内部 QA のあと、`policy_items` / `verification_items` の要点を `implementation_items` に統合したうえで組み立てる。**ブロックA【描画内容】** にそのまま差し込む。**全文日本語**（短い技術識別子は可）。

### 組み立て順序

1. `compressed_summary` を組み立て、抽出内容の整合を確認する
2. `policy_items` / `verification_items` の要点を `implementation_items` にマージする（画像プロンプトへ直接書かない）
3. マージ後の `*_items` から `image_generation_prompt` を生成する

### 形式ルール

- **プレーンテキストのみ**（`##` / markdown 表 / 入れ子 fenced 禁止）
- `image_generation_prompt` 単体で **2000〜2800 字**
- 画像内に描画する固定文言は **「」で囲む**（タイトル・キャプション・コールアウト・熊のセリフ）
- まとまり見出し: `【1.背景】` / `【2.目標】` / `【3.実装】`（必須）、`【4.残課題】`（**任意**。`remaining_items` が空なら見出しごと省略）
- 先頭: `タイトル: 「{title}」`
- **各列に `列リード:` 1 文（50〜90 字）を必須**
- 各項目: `- ミニ図解: {内容の説明}。キャプション「{名詞+説明文}」` 形式（背景・目標は **各 3 項目**）
- キャプションは **70〜110 字/項目**（技術前提・影響を含む名詞+説明文）
- 実装列: **小見出し帯**（`■ {帯名}`）を **2 つ以上**、`- 表: {表の内容}` **1 つ以上**、`- フロー: {手順}` **1 つ以上**、`- 黄コールアウト「{短文}」` を **2 つ以上**
- 末尾: `熊の吹き出し「{mascot_line}」`
- `remaining_items` が空のときは `【4.残課題】` セクションを **出力しない**（`（なし）` 行も不要）

### 記述例（構造のみ）

```text
タイトル: 「{title}」

【1.背景】
列リード: {現状と課題を50〜90字で。なぜ今やるか・制約を含む}
- ミニ図解: フロント静的列定義とREST/PATCH APIが分断された図。赤×で接続を示す。キャプション「一覧・詳細・作成がバラバラ — 型共有と保守が困難」
- ミニ図解: 人事台帳・取引先台帳はSDUI+oRPC済み、備品だけ旧方式の対比図。キャプション「他ドメインはSDUI移行済み — 備品個体だけ旧静的UI+フラットPATCHが残存」
- ミニ図解: 属性定義がフロントとAPIで二重管理されている図。キャプション「列定義のSSOTがなくフロントとAPIで二重管理 — 変更時の不整合リスク」

【2.目標】
列リード: {達成状態を50〜90字で。Phase 0の成功条件を含む}
- ミニ図解: チェックリストと金バッジ。キャプション「Phase 0 — OpenSpec設計・oRPC契約骨格・属性マトリクスを確定」
- ミニ図解: SSOTからoRPC APIへの橋。キャプション「list/detail/create/profileの4 capability → oRPC 6本に整理」
- ミニ図解: 設計書フォルダに緑チェック。キャプション「実装前に設計を固定 — BE/FEはPhase 1以降の土台を先に固める」

【3.実装】
列リード: {本PR/本ADRの実装要点を50〜90字で。ドキュメントのみ等のスコープも明示}
■ capability spec
- ミニ図解: 3面グリッド（default/person/inventory）。キャプション「equipment-list — 3 viewMode対応のDataGrid仕様」
- ミニ図解: 7ステップウィザード。キャプション「equipment-create — 7ステップの作成フロー仕様を定義」
■ 設計・契約骨格
- 表: 25列の属性マトリクス。fieldKey・viewMode・inventoryListVisible列を含む
- フロー: GET baseline → sections PUT → 204 → 再GET の保存契約
- 黄コールアウト「D8 fieldKey確定 — 25列の属性キーを設計で固定」
- 黄コールアウト「D9 複合属性の扱い — 全6 APIで認可・テナント分離を明記」

【4.残課題】
列リード: {スコープ外・後続フェーズを50〜90字で}
- ミニ図解: 工事バリアとPhase 1矢印。キャプション「BE/FE実装コードは含まない — Phase 1以降で着手」

熊の吹き出し「{mascot_line}」
```

## 取得方法（入力の形だけ判別）

| 入力の形               | 取得                                                                                                                                                           |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `github.com/.../pull/` | `gh pr view <URL> --json title,body,commits,files,headRefName,baseRefName,statusCheckRollup` と `gh pr diff <URL>`（大きいとき `--stat` 後に主要ファイルのみ） |
| gh 不可                | GitHub MCP `pull_request_read`（get / get_diff / get_files）                                                                                                   |
| `.md` ファイルパス     | Read で全文                                                                                                                                                    |
| それ以外のテキスト     | 引数＋会話中の `@` 添付を Read                                                                                                                                 |

複数入力はすべて読み、**1 枚に統合**（4 列合計が 12 を超え、自動圧縮で伝えられなくなる場合のみ、ユーザーに優先ソースを 1 点確認）。

## 抽出の焦点（ソース共通）

- **背景**: なぜ今やるか・現状・制約・課題の前提
- **目標**: 何を達成するか・成功状態・ユーザー影響
- **実装**: 採用案・設計・実装の要点・検証（**最も厚く書く**）
- **残課題**: 未完了・後続 PR・フォロー（**任意**。なければセクション省略）
- **避ける**: diff 行コピペ、パス一覧、縦積みレイアウト向けの手順列

## ソース別の補足（取得後の読み方のみ）

**PR**: 背景=課題・文脈、目標=PR の目的、実装=変更内容・トレードオフ・テスト、残課題=未マージの follow-up・TODO。方針=受け入れ条件・非機能要件の要約。

**ADR**: 背景=コンテキスト、目標=決定で満たす要件、実装=決定内容と影響、残課題=未決・移行の残り。

**開発ログ**: 見出し「概要/分析」→背景・目標、「方針」→`## 方針`、「実装/検証」→`## 実装`（厚く）、残課題→`## 残課題`。

**自由記述**: ユーザーが伝えたい Before/背景・After/目標・やったこと/実装・残りを 4 列に割り当て。不足時は推測せず 1〜2 点確認。

## mascot_line

- 20 字前後、1 文、**ユーモア可**（技術テーマに関連した比喩・語呂が望ましい）
- `image_generation_prompt` 末尾の `熊の吹き出し「」` に入れる（`compressed_summary` には追記しない）
- 例: 「キャッシュは生もの！ちゃんと管理しないと腐るクマー！」

## 機密・品質

- 個人情報・認証情報・内部 URL の機密部分は除外
- `compressed_summary` は内部 QA 用のため読みやすさ優先で **過度に短くしすぎない**（実装列は ADR-369 サマリ程度の密度を目安）
- `image_generation_prompt` は **2000〜2800 字**、**全文日本語**、各列 **列リード必須（50〜90字）**、実装列に **表+フロー必須**、背景・目標に **ミニ図解各3項目**、キャプション **70〜110字/項目**、画像内固定文言は **「」** で囲む
- diff だけでは要点が分からない場合のみ、コードベースの補足調査可（任意）
