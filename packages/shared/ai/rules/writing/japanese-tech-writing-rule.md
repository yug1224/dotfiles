# 日本語技術文書の文章規範

日本語の技術原稿（書籍の章、記事、解説文）を書く・推敲するときに従う。LLM 向けには、禁止リストと執筆後点検を優先する。

## 整形

- 一文ごとに改行する。段落の区切りは空行で示す。
- コード、差分、ログ、設定断片はコードブロックで示す。
- 定義・分類の列挙は箇条書きでよい。定義される用語・初出の導入語は太字。以後の言及・引用・通称は「」に使い分ける。
- 用語と定義の箇条書きは「**用語**：説明」（全角コロン）。区切り線でつないでない。
- ダッシュ（`—` `―` 「——」）を地の文・見出しで使わない。同格・補足は（）、言い換えは句点で二文にするか読点でつなぐ。en ダッシュ・英語複合語・コード・書誌は対象外。
- 中黒（・）を日本語の並列で使わない（固有名詞の内部は可）。
- 見出しに区切り線で二要素を詰め込まない。単一の自然な句にする。コラム見出しも種別名だけで済ませず、内容を特定する。

**技術書モード**: 本筋から外れる補足（用語の由来、定式化名など）は脚注（`[^ラベル]`）へ降ろす。記事・ブログでは脚注細部は必須としない。

## 段落と論証の構成

パラグラフライティングを基本とする。段落は論証の一歩である。

- 一段落一トピック。場面の進行が混ざる長い段落は分割する。
- 段落の最初の文で、何の話かがわかるようにする。
- 段落先頭で前段落との論理関係を接続表現で明示する（「であれば」「実際」「しかし」など）。
- 新概念は辞書型断定で始めない。対象 → 働き・差分 →（必要なら）定義、の順。
- 論証は一方向。反論・疑念を処理してから結論を一度だけ置く。結論後に反論を戻さない。
- 例への弁明は山場直後に挟まず、次節冒頭でまとめる。
- 誤った解釈は明示的に否定してから本当の理由を述べる。「AではなくB」には否定の根拠を一文添える。
- 譲歩は事実確認にとどめる。あとで訂正する内容を著者の声で断定しない。通説・読者の声に帰属させる。
- 山場で効かせたい情報は手前で先出ししない。前方参照は論証が一段落した位置に置く。
- 否定・限定する命題は「」で正確に書き出す。漠然とした否定で済ませない。

## 論証の厳密さ

- 推量・可能性・疑念・反実仮想を、根拠なく断定へ変えない。「かもしれない」等は根拠なき弱めのときだけ削る。
- 異なるものを「同じ」とまとめない。複数要因を単一原因に還元しない。
- 章・節をまたいで分類・定義・用語の地位を一致させる。
- 因果には機構を一文で示す。「必ず」検出・保証・解決できるかのように書かない。
- 例が主張の全体を支えないなら、主張の範囲を狭める。前方参照は必ず回収する。
- 譲歩・限定のあとに論を進め、逆接で宙吊りにしない。中心語は定義・範囲のあとで使う。
- 複数概念を上位語にまとめる直前に、帰着の一文を置く。

## 読み手の負荷・視点・演出

- 後で参照しない固有名を出さない。抽象語が一意でないときは括弧の同格でその場で特定する。
- 新しい例を足すときは、前の例との差分と必要性を前置きする。問い・帰結に無関係な過剰詳細は omit する。
- 例示は行為者を主語にした動作の連なりで書く。架空の人物設定を無意味に冠しない。
- 論証中は読者を「あなた」と呼ばず役割名で書く。二人称は導入・結びの要所に限る。
- 対象語は具体に選ぶ。導入した術語に後退しない。慣用の術語・訳語を使い、術語っぽい語を非術語場面に流用しない。
- 演出は節度。溜め・修辞疑問・決め台詞の独立段落・太字強調・劇的転回・危険の煽りは要所に限る。「重要なのは〜」で予告しない。慣用のひねりや曖昧な比喩を使わない。

## LLM っぽい表現の禁止（優先点検）

論点を増やさず「ちゃんと書いている感」だけを付ける型を使わない。

- **予告と総括**: 「重要なのは〜」「本章では〜を扱う／探求する」「ここでは〜について見ていく」「まとめると」「要するに」（直前の言い換えだけ）、「〜に他ならない」
- **正面から系**: 「正面から扱う／回収する／見る／書く／立てる」
- **空虚な形容**: 「不可欠」「核心的」「鍵となる」「根本的な」（中身なし強調）、「多角的」「包括的」「総合的」
- **空虚な動詞**: 「掘り下げる」「深掘りする」「言語化する」「触れる」「言及する」
- **接続の型**: 「〜において」「〜という側面から」「〜の観点から」（新情報なし）、「さらに」「また」「加えて」の連打
- **弱い緩和と空虚な強調**: 根拠なき「〜と言えるだろう」「〜かもしれない」、「非常に」「極めて」「大いに」
- **機械的な装飾**（参考）: 誇張（「革命的」「ゲームチェンジャー」）、機械的な太字強調の連打、絵文字付きリストの装飾列挙

良い置換の方向: 姿勢の宣言を削り、扱う対象・回収する前提・評価の中身を直接書く。

## 冗長の排除・見出し・誠実さ

- 同じ主張の言い換え繰り返し、隣接節の役割重複、場面直後の要約し直しをしない。意味づけの一文だけ置く。
- 読者が補える中間段階、接続・評価だけの文、想像上の問答、メタな枠取りでの発想紹介、著者立場の弁明を置かない。
- 未導入の概念・文書名を先出ししない。根拠で確定していることは具体的に言い切る（不確実性・仮定・疑念の弱い述語は保持）。
- 見出しは節が答える問い、または扱う対象を指す。手順だけの見出し・情報ゼロ・結論セリフ化を避ける。
- 作為的に見えうる例は隠し、読者の経験・通説に訴えて根拠を短く添える。未確認を確認済みのように書かない。

## 執筆後点検（機械的）

1. 禁止リスト（LLM 口調・ダッシュ・中黒並列・見出し区切り線）を検索して削る／直す。
2. 段落ごとに「受けるもの／役割／渡すもの」が書けない段落をギャップ候補にする。
3. 推量の断定化、同物まとめ、単一原因還元、未回収の前方参照を潰す。
4. 同一主張の重複、場面直後の要約、弁明・断りを削る。
5. 見出しが問い／対象を指しているか確認する。

## 適用スライス

コマンドや媒体ルールは、次のいずれかを指して部分適用する。指定がなければ `full`。

| スライス        | 適用範囲                                                                                                                                              | 典型の呼び出し元                                                                                                 |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `full`          | 本ファイル全体                                                                                                                                        | 技術書・技術記事の執筆・推敲（スライス未指定時の既定）                                                           |
| `blog-base`     | `writing-style-rule` の Override 表と同ファイル「共通適用」のみ。JTW の整形・段落論証・脚注の機械適用はしない。Override の詳細は writing-style が正本 | `writing-style-rule` / `plan-blog` 経由                                                                          |
| `tech-doc-lite` | 「LLM っぽい表現の禁止」と、冗長・根拠なき断言の抑制のみ。一文一行・段落論証の機械適用はしない                                                        | `suggest-development-log` / `suggest-pr-description`（採用メッセージ系 `.local` も同スライスを明示する場合あり） |
