# テスト戦略マトリクス

テスト種別の使い分け基準と設計指針の一覧。

## テストピラミッド

```
        ┌─────────┐
        │  E2E    │  少数 — ユーザーフロー単位
        ├─────────┤
        │  統合   │  中程度 — モジュール連携・DB アクセス
        ├─────────┤
        │  単体   │  多数 — ロジック単位
        └─────────┘
```

## テスト種別の使い分け

| 種別                     | 対象                           | 速度 | 信頼性 | 使用場面                                             |
| ------------------------ | ------------------------------ | ---- | ------ | ---------------------------------------------------- |
| **単体テスト**           | 純粋関数、ユースケースロジック | 高速 | 高     | ビジネスルール、計算ロジック、状態遷移               |
| **統合テスト**           | モジュール間連携、DB クエリ    | 中速 | 中     | Repository の CRUD、API ハンドラー、ミドルウェア連携 |
| **E2E テスト**           | ユーザーフロー全体             | 低速 | 低     | 重要な業務フロー、決済、認証フロー                   |
| **コンポーネントテスト** | UI コンポーネント単体          | 高速 | 高     | 表示ロジック、インタラクション、状態変化             |
| **Visual Regression**    | UI の見た目                    | 中速 | 中     | デザイン変更の検出、テーマ切り替え                   |

## レイヤー別テストパターン

### バックエンド

| レイヤー             | テスト種別  | ファイル命名    | モック対象               |
| -------------------- | ----------- | --------------- | ------------------------ |
| UseCase / Service    | 単体        | `*.spec.ts`     | Repository, 外部サービス |
| Repository / Query   | 統合（SQL） | `*.sql.spec.ts` | なし（実 DB 使用）       |
| Controller / Handler | 統合（API） | `*.api.spec.ts` | 外部サービスのみ         |
| Guard / Middleware   | 単体        | `*.spec.ts`     | リクエストコンテキスト   |

### フロントエンド

| レイヤー       | テスト種別 | ファイル命名    | モック対象       |
| -------------- | ---------- | --------------- | ---------------- |
| ユーティリティ | 単体       | `*.spec.ts`     | なし             |
| Hooks          | 単体       | `*.spec.ts`     | API クライアント |
| コンポーネント | Storybook  | `*.stories.tsx` | Props / Context  |
| ページ         | E2E        | `*.e2e.ts`      | なし（実環境）   |

## テストケース設計の観点

### 正常系

- 期待する入力で期待する結果が返るか
- 複数の正常パターンをカバーしているか

### 異常系

- 無効な入力（null, undefined, 空文字, 負数）
- 認証・認可エラー
- 外部サービスのエラーレスポンス
- タイムアウト / ネットワークエラー

### 境界値

- 最小値 / 最大値 / ちょうど境界
- 空配列 / 1要素 / 上限要素数
- 文字列長の上限 / 下限

### エッジケース

- 並行実行 / 競合状態
- 冪等性（同じ操作を複数回実行）
- 大量データ

## モック戦略

| 依存先       | 推奨アプローチ      | 理由                                   |
| ------------ | ------------------- | -------------------------------------- |
| 外部 API     | MSW / nock でモック | 外部サービスの不安定さを排除           |
| データベース | 統合テストでは実 DB | モックだとクエリの正確性が検証できない |
| 時間（Date） | vi.useFakeTimers    | 時間依存のロジックを安定テスト         |
| ファイル I/O | vi.mock / memfs     | テスト間の干渉を防止                   |
| 環境変数     | vi.stubEnv          | 環境依存の分岐をテスト                 |

## テストデータ管理

| パターン    | 使用場面           | 特徴                                        |
| ----------- | ------------------ | ------------------------------------------- |
| **Factory** | 単体テスト         | テストごとに新規生成、柔軟にカスタマイズ    |
| **Fixture** | 統合テスト         | 共通のシードデータ、DB リセットと組み合わせ |
| **Builder** | 複雑なオブジェクト | メソッドチェーンで必要なフィールドだけ指定  |

## カバレッジ目標の目安

| 対象                        | 推奨カバレッジ | 根拠                         |
| --------------------------- | -------------- | ---------------------------- |
| ビジネスロジック（UseCase） | 90%+           | バグの影響が最も大きい       |
| ユーティリティ関数          | 90%+           | 再利用が多く、影響範囲が広い |
| API ハンドラー              | 80%+           | 正常系 + 主要な異常系        |
| UI コンポーネント           | 70%+           | Storybook でカバー           |
| インフラ / 設定             | 50%+           | スナップショットテスト中心   |
