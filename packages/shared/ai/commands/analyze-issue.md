チケット URL または ID で指定された課題（QA指摘・バグ報告・障害情報等）の内容を読み取り、コードベースを調査して原因を特定し、松竹梅の対応方針を提案する。**調査・提案のみ**であり、ユーザーが明示しない限り自動修正しない。

**Input**: `/analyze-issue` の後に続く引数はチケット URL またはチケット ID（必須）。取得方法は `ticket-retrieval-rule`（および存在すれば `.local.md`）に従う。

**使用例**:

- `/analyze-issue https://github.com/owner/repo/issues/123`
- `/analyze-issue TICKET-1234`

---

## 参照ドキュメント

### ルール

1. **`@~/.config/shared/ai/rules/conventions/ticket-retrieval-rule.md`** — チケット取得（必須）
2. 同ディレクトリの **`ticket-retrieval-rule.local.md`** — プロバイダ固有アダプタ（存在時のみ）
3. **`@~/.config/shared/ai/rules/conventions/coding-rule.local.md`** — コーディング規約（Glob で存在する場合のみ・任意）

### Advisor（深掘り観点）

課題の性質に応じて、以下の Advisor エージェントを選択的に活用する:

- **`design-advisor`**: アーキテクチャ・API 設計・DB スキーマ・ドメインモデリング関連の原因分析
- **`build-advisor`**: FE/BE 実装パターン・インフラ構成・アクセシビリティ関連の原因分析
- **`quality-advisor`**: セキュリティ・テスト不足・品質劣化・パフォーマンス関連の原因分析

## Steps

### 0. トレース（必須）

応答の冒頭に `✅️: /analyze-issue` と出力する。

### 1. チケット取得ルールの読み込み（必須）

1. `@~/.config/shared/ai/rules/conventions/ticket-retrieval-rule.md` を Read（必須）
2. 同ディレクトリの `ticket-retrieval-rule.local.md` を Glob。存在する場合のみ Read（矛盾時は `.local.md` を優先）

### 2. コーディングルール（任意）

`@~/.config/shared/ai/rules/conventions/coding-rule.local.md` を Glob で確認する。

- **存在する場合のみ** Read し、サブエージェント方針・コーディング規約を把握する
- **存在しない場合はスキップ**（必須にしない。読み込み失敗で中断しない）

### 3. チケット情報の取得

引数からチケット URL または ID を取得する。未指定の場合はユーザーに入力を求める。

Step 1 のアダプタに従いチケット情報を取得する（タイトル・目的・受け入れ条件など。詳細は `ticket-retrieval-rule`）。

取得内容を分析し、以下を整理してユーザーに共有し、調査方向性の認識を合わせる:

- **課題の要約**: 何が問題か（1-2文で簡潔に）
- **影響範囲**: どの機能・画面・モジュールに影響するか
- **再現条件**: どうすれば再現できるか（記載がある場合）
- **調査の方向性**: どのコード領域を重点的に調べるべきか

### 4. 原因調査

以下を一括で収集したうえで、サブエージェント（Task ツール）で多角的に調査する。
複数の独立した調査対象がある場合は並列起動する（最大4つ）。**いずれも `readonly: true`**。この Step ではコードを変更しない。

**a. コンテキスト収集**

- 対象リポジトリの `.cursor/rules/`（または同等）を動的探索し、関連ルールを Read する
- `analyze-issue.local.md` を Glob。存在する場合のみ Read し、追加観点を適用する

**b. コードベース調査（`explore`, `readonly: true`）**

- `.codegraph/` がある場合、構造・フロー調査は `@~/.config/shared/ai/rules/conventions/token-optimization-rule.md` に従い CodeGraph を explore / Grep より先に使う
- 課題に関連するコード・モジュールの特定
- エラーメッセージやログ出力箇所からの逆引き
- 関連する最近のコミット・変更履歴の確認（`git log` で変更経緯を追跡）

**c. 深掘り調査（Advisor, `readonly: true`）**

課題の性質に応じて選択的に起動する（`subagent_type` で指定）:

- 設計・API・DB 関連: `design-advisor`
- 実装・FE/BE・インフラ関連: `build-advisor`
- セキュリティ・テスト・品質関連: `quality-advisor`

複数の観点にまたがる場合は、該当する Advisor を並列で起動する。

### 5. 対応方針の提案（松竹梅）

調査結果を統合し、対応方針の**下書き**を作成する（この時点ではユーザーに提示しない）:

- **原因の特定結果**: 根本原因と直接原因を区別して記載
- **影響範囲の評価**: 調査で判明した影響範囲（Step 3 の想定との差分も記載）
- **松竹梅の対応方針**:
  - **松（理想的な対応）**: 根本原因を解消し、再発防止まで含めた対応
  - **竹（推奨の対応）**: 直接原因を修正し、実用的なバランスの対応
  - **梅（最小限の対応）**: 最小限の変更で課題を解消する対応
- 各方針について、変更対象ファイル・リスク・想定工数を明記する

`@~/.config/shared/ai/rules/conventions/output-verification-rule.md` を Read し、**「インライン再検証」**に従って下書きを検証・修正したうえで、下記出力フォーマットでユーザーに提示する。

**自動修正は行わない。** 実装はユーザーが方針を選んで依頼したときのみ。

## 出力フォーマット

Markdown 形式で以下の全セクションを含める:

- **課題サマリー**: チケット内容を簡潔に要約
- **調査結果**:
  - 根本原因
  - 直接原因
  - 関連コード（ファイルパス + 該当箇所の引用、`file.ts:42` 形式）
- **影響範囲**: 影響を受けるモジュール・機能の一覧
- **対応方針（松竹梅）**: 3段階の対応案
  - 各案の内容
  - 変更対象ファイル
  - リスク
  - 想定工数
- **推奨**: どの方針を推奨するか + 理由

原因が特定できなかった場合も、調査した内容と推測を明示し、「追加調査の方向性」セクションを追加する。

## Guardrails

- チケット URL / ID が未指定の場合はユーザーに入力を求める
- チケット取得は `ticket-retrieval-rule`（+ 存在すれば `.local.md`）に従う。外部チケット固有手順を本コマンド本文にハードコードしない
- `coding-rule.local.md` は任意。欠如を理由に中断しない
- サブエージェントは `readonly: true` で安全に調査する
- 調査中にコードの変更は行わない（調査と提案のみ）。修正はユーザー依頼時のみ
- 原因が特定できない場合は、調査内容と推測を明示し、追加調査の方向性を提案する
- 対象リポジトリのルール照合は Step 4 で1度だけ行い、以降のステップで一貫して適用する
- Step 3 で調査計画をユーザーに共有し、方向性の認識を合わせてから調査に進む
- 最終 Step の再検証完了前に、成果物をユーザーへ出力しない
