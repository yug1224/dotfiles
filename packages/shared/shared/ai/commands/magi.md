MAGI System の 3 ユニット（MELCHIOR-1, BALTHASAR-2, CASPER-3）を並列投入し、異なる視点からの合議判定を行う。

**Input**: `/magi` の後に続く引数は、評価対象の議題（必須）。`@` によるファイル添付や会話中のコンテキストも考慮する。

**使用例**:

- `/magi React Server Components を導入すべきか`
- `/magi モノレポ構成を維持するか、リポジトリ分割するか @README.md`
- `/magi CI/CD パイプラインに E2E テストを追加する方針の是非`

---

## Steps

### 0. トレース（必須）

応答の冒頭に `Applied: /magi` と出力する。

### 1. 開始メッセージの出力

`--- MAGI System 起動: 3 ユニット並列解析中 ---` と出力する。

### 2. 議題の整理

引数・添付ファイル・会話コンテキストから、評価対象の議題を明確化する。
議題が不明確な場合はユーザーに質問する。

不明確と判断する基準（以下のいずれかに該当する場合）:

- 意思決定の問い（何を判断したいか）が含まれていない
- 背景情報がなく前提条件が推定できない
- 選択肢が多すぎて議題タイプの分類が困難

整理すべき情報:

- 何を判断したいのか（意思決定の問い）
- 背景・制約条件
- 想定している選択肢（あれば）
- **影響度**: 大/中/小（各ユニットの分析深度のガイドとする）
- **領域**: 技術/プロセス/戦略/その他（各ユニットが分析観点の重み付けに使用する）
- **議題タイプ**: 以下のいずれかに分類する
  - **Go/No-Go 型**: やるかやらないかの判断
  - **二択型**: A or B の選択（比較分析を重視、各ユニットは推奨案を明示する）
  - **設計型**: どう実装するかの判断（設計提案を重視する）
  - **優先度型**: 何を先にやるかの判断（トレードオフ分析を重視する）
  - **カスタム型**: 上記に当てはまらない議題。ユーザー指定の評価観点に従うか、オーケストレーターが最適な評価観点を判断する

### 3. 3 体を並列投入

Task ツールで以下の 3 つの Subagent を **1つのメッセージ内で同時に** 起動する（逐次起動しない）。

- **MELCHIOR-1**: `subagent_type`: `"melchior-1"`, `readonly: true`
- **BALTHASAR-2**: `subagent_type`: `"balthasar-2"`, `readonly: true`
- **CASPER-3**: `subagent_type`: `"casper-3"`, `readonly: true`

エージェント定義は Cursor がサブエージェントファイルから自動ロードするため、プロンプトには議題に必要な情報のみを含める。各ユニットへのプロンプトは以下のテンプレートに従う:

```
## MAGI 合議ルール

- 他のユニットの判断は知らない前提で、独立して分析する
- 日本語で回答する（技術用語は英語可）
- 議題がコードベースに関わる場合は、検索・読み取りツールで調査し、具体的な根拠を示す
- 分析は必ず具体的な根拠（コード、数値、事例）を伴うこと。根拠のない主張は避ける
- 「影響度」と「領域」に応じて分析の深度と重み付けを調整すること
- 他の2体と意見が一致するであろう部分は簡潔に、自分のペルソナ固有の視点を厚く述べること

## 確信度の判定基準

`packages/shared/shared/ai/rules/magi/magi-unit-common-rule.md`（mise-dotfiles 後 `~/.config/shared/ai/rules/` 等。Cursor ラッパー: `magi-unit-common-rule.mdc`）の基準に従う。

## 議題タイプ別の出力ガイド

- **二択型**: 判定に加えて、推奨案（A or B）とその理由を評価の冒頭で明示すること
- **設計型**: 判定に加えて、推奨する設計アプローチを「条件・提言」セクション内の冒頭で具体的に提案すること
- **優先度型**: 判定に加えて、推奨する優先順位とその根拠を評価の冒頭で明示すること
- **カスタム型**: ユーザー指定の評価観点がある場合はそれに従い、ない場合は自身のペルソナに基づく分析観点で評価すること
```

### 4. 回答の収集・合議判定・交差分析

`packages/shared/shared/ai/rules/magi/magi-orchestration-rule.md`（mise-dotfiles 後 `~/.config/shared/ai/rules/` 等。Cursor ラッパー: `magi-orchestration-rule.mdc`）に従い、以下を実施する:

1. 回答の収集と成功/失敗判定
2. 判定・確信度の抽出
3. 縮退運用（必要な場合）
4. 合議判定
5. Devil's Advocate（全会一致の場合）
6. 交差分析

### 5. MAGI レポートの下書き作成

下記フォーマットに従いレポートの**下書き**を作成する（この Step ではユーザーに提示しない）。各ユニットの判定と確信度はオーケストレーターが回答から読み取ってヘッダーに転記し、評価・根拠・条件・提言は要約せずそのまま掲載する。

### 6. 出力の再検証（必須）

`@~/.config/shared/ai/rules/conventions/output-verification-rule.md` を Read し、**「インライン再検証」**に従って下書きを検証・修正する。

### 7. 最終レポート出力

Step 6 で修正した MAGI レポートのみをユーザーに提示する。

## Guardrails

- 議題が未指定・不明確な場合はユーザーに質問する
- 3 体は必ず **並列** で起動する（逐次実行しない）
- 各ユニットの判定・確信度・一言サマリーはオーケストレーターがヘッダーに転記し、評価・根拠・条件・提言は要約せずそのまま掲載する
- 合議判定では `packages/shared/shared/ai/rules/magi/magi-orchestration-rule.md`（Cursor ラッパー: `magi-orchestration-rule.mdc`）の確信度重み付けルールに従う
- 全会一致の場合は Devil's Advocate 注記を必ず付記する
- 全会一致でない場合は、少数意見の根拠を特に強調して提示する
- 松竹梅のアクション提案は具体的かつ実行可能な内容にする
- 新しい議題タイプを追加する場合は、Step 2 の議題タイプ一覧、Step 3 の議題タイプ別出力ガイド、および影響度・領域・議題タイプのテンプレート変数の 3 箇所を更新すること
- 最終 Step の再検証完了前に、成果物をユーザーへ出力しない
