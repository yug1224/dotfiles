要件（チケット / GitHub Issue / 自然言語）を起点に、設計・実装の両面から分析し Must have / Nice to have の実装計画を策定する。ユーザーが「サブエージェントを使って」「マルチエージェントで」等とサブエージェント利用を明示している場合のみ、Task ツールで `design-advisor` と `build-advisor` を起動する。明示がない場合は親エージェントが Step 3 の観点に沿って直接分析する。全ステップが読み取り専用のため、Plan モードでの実行を推奨する。

**参照ルール**: `@~/.config/shared/ai/rules/checklists/nfr-checklist.md`（Cursor ラッパー: `packages/cursor/rules/checklists/nfr-checklist.mdc`）

**Input**: `/suggest-plan` の後に続く引数は、要件の説明文、チケット ID またはチケット URL（任意）。取得方法は `ticket-retrieval-rule`（および存在すれば `.local.md`）に従う。

**使用例**:

- `/suggest-plan ユーザー一覧画面にフィルタリング機能を追加`
- `/suggest-plan PROJ-1234`
- `/suggest-plan https://github.com/owner/repo/issues/123`

---

## Steps

### 0. トレース（必須）

応答の冒頭に `Applied: /suggest-plan` と出力する。

### 1. 要件の収集

以下の情報源を**並列で**収集する。

a. **チケット情報**（引数がある場合）

`@~/.config/shared/ai/rules/conventions/ticket-retrieval-rule.md` に従い、チケットの目的・背景・タイトル・受け入れ条件を取得する。同ディレクトリの `ticket-retrieval-rule.local.md` があれば併せて Read（無ければスキップ）。

b. **コンテキスト情報**

ユーザーが `@` で添付したファイルや会話中で提供した情報があれば、それも考慮する。

c. **プロジェクト構造**

ワークスペースルートで以下を実行し、プロジェクトの技術スタックとディレクトリ構成を把握する。`.cursor/rules/` 内の関連ルールがあれば Read で確認する。

```bash
ls -la
```

### 2. 要件の整理と確認

収集した情報を以下の形式で整理し、ユーザーに共有する:

- **目的**: 何を実現するか（1-2文）
- **スコープ**: 変更が必要な領域（FE / BE / DB / インフラ等）
- **制約**: 技術的制約・期限・互換性要件等

方向性に認識齟齬がないか確認してから、次のステップに進む。

### 3. 設計・実装分析

サブエージェント利用の判定:

1. 同一メッセージまたは直前の会話で「サブエージェントを使って」「マルチエージェントで」等の明示がある → 下記「サブエージェントモード」で Task 起動
2. 明示がない → 親エージェントが **a.** / **b.** を直接実施
3. 判断できない → ユーザーにサブエージェントを使うか確認してから 1 または 2 を実行

以下の観点で設計面・実装面の両面から分析する。

**a. 設計分析**

- アーキテクチャ上の影響（レイヤー構成、モジュール間依存）
- API 設計（新規エンドポイント / 既存変更の必要性）
- DB スキーマ変更の必要性（マイグレーション要否）
- 非機能要件への影響（`@~/.config/shared/ai/rules/checklists/nfr-checklist.md` の観点）

**b. 実装分析**

- 既存の実装パターンとの整合性
- 変更対象ファイルの特定と変更内容の概算
- テスト戦略（追加・更新が必要なテスト）
- 実装上のリスクと注意点

**サブエージェントモード**（判定リストの 1 が成り立つ場合にのみ以下を適用）

Task ツールで以下の 2 つの Subagent を**1つのメッセージ内で同時に**起動する（逐次起動しない）。

- **設計分析**: `subagent_type`: `"design-advisor"`, `readonly: true`
- **実装分析**: `subagent_type`: `"build-advisor"`, `readonly: true`

サブエージェントへのプロンプトには、Step 1-2 で収集した要件・プロジェクト構造と上記の分析観点を含める。

### 4. Must have / Nice to have の実装計画を策定

Step 3 の分析結果を統合し、以下の 2 段階で計画を策定する。

- **Must have**: 要件を満たし主要な品質を担保する実用的な実装
- **Nice to have**: 設計品質・テスト・非機能要件をさらに高める追加施策

### 5. 実装計画の下書き作成

下記フォーマットに従い**下書き**を作成する（この Step ではユーザーに提示しない）。

### 6. 出力の再検証（必須）

`@~/.config/shared/ai/rules/conventions/output-verification-rule.md` を Read し、**「インライン再検証」**に従って下書きを検証・修正する。

### 7. 最終出力

Step 6 で修正した内容を、下記フォーマットに従ってユーザーに提示する。

---

## 出力フォーマット

```markdown
## 実装計画

### 要件サマリー

（目的・スコープ・制約の要約）

### 設計概要

（mermaid 図を含むアーキテクチャ・データフローの概要）

### 変更対象

| ファイル / モジュール | 変更種別           | 概要           |
| --------------------- | ------------------ | -------------- |
| `path/to/file.ts`     | 新規 / 修正 / 削除 | 変更内容の概要 |

### Must have

- 実装内容の詳細
- 追加テスト
- 想定工数

### Nice to have

- 追加施策の詳細
- 追加テスト
- 想定工数

### リスクと注意点

- リスク1
- リスク2
```

## Guardrails

- `/suggest-plan` は調査と計画策定のみを行い、コードの変更は行わない
- サブエージェント利用の明示が読み取れない場合は、Step 3 に進む前にユーザーへ確認する
- サブエージェントモード時は Advisor を `readonly: true` で起動する
- 要件が曖昧な場合は Step 2 でユーザーに確認してから先に進む
- 出力には必ず mermaid 図を含め、設計の全体像を視覚化する
- 各プランには変更対象ファイルと想定工数を明記する
- 既存の実装パターン・ルールとの一貫性を最優先にする
- チケット情報の取得は引数で ID/URL が渡された場合のみ実行し、渡されなければスキップする
- 最終 Step の再検証完了前に、成果物をユーザーへ出力しない
