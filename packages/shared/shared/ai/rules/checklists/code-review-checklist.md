# コードレビュー観点チェックリスト

PR / コード変更のレビュー時に確認すべき観点の一覧。

## 正確性

- [ ] ビジネス要件を正しく実装しているか
- [ ] エッジケース（null, 空配列, 境界値）が考慮されているか
- [ ] エラーハンドリングが適切か（握りつぶしていないか）
- [ ] 並行処理の安全性が確保されているか（競合状態、デッドロック）

## 設計・アーキテクチャ

- [ ] レイヤー境界を越えた依存がないか
- [ ] 単一責任の原則に従っているか
- [ ] 適切な抽象レベルで設計されているか（過剰設計 / 設計不足）
- [ ] 既存パターンとの一貫性があるか
- [ ] モジュール間の依存方向が正しいか

## セキュリティ

- [ ] SQL / NoSQL インジェクションの可能性がないか
- [ ] 認証・認可の漏れがないか（ガード / ミドルウェアの適用忘れ）
- [ ] 秘密情報（API キー、トークン）がハードコードされていないか
- [ ] 入力バリデーションが十分か
- [ ] XSS / CSRF の対策がされているか

## パフォーマンス

- [ ] N+1 クエリの可能性がないか
- [ ] 不要な再レンダリングがないか（React 依存配列、メモ化）
- [ ] インデックスのないクエリがないか
- [ ] 大量データの一括処理は適切か（バッチ / ストリーミング）
- [ ] 不要な API 呼び出しがないか

## 可読性・保守性

- [ ] 命名が明確か（`data`, `info`, `manager` 等の曖昧な名前を避ける）
- [ ] 関数の長さが適切か（1関数1責務）
- [ ] マジックナンバー / マジックストリングがないか
- [ ] 重複コードがないか（DRY）
- [ ] 複雑な条件分岐が整理されているか

## テスト

- [ ] テストが追加・更新されているか
- [ ] 正常系・異常系・境界値がカバーされているか
- [ ] テストが実装詳細に結合していないか
- [ ] テスト名が何をテストしているか明確か

## ドキュメント・コミュニケーション

- [ ] PR の説明が変更内容を適切に伝えているか
- [ ] 複雑なロジックにコメントがあるか（Why を説明）
- [ ] API の変更がある場合、ドキュメントが更新されているか
- [ ] Breaking change がある場合、移行手順が示されているか

## 重要度の判断基準

| ラベル         | 基準                                       | 対応       |
| -------------- | ------------------------------------------ | ---------- |
| **Critical**   | バグ、セキュリティ脆弱性、データ損失リスク | 必ず修正   |
| **Suggestion** | 設計改善、可読性向上、ベストプラクティス   | 検討を推奨 |
| **Nit**        | スタイル、命名の微調整、好みの範囲         | 対応は任意 |
