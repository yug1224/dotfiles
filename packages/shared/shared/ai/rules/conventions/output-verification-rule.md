応答の冒頭に「Applied: output-verification-rule」と出力する。

# 出力再検証ルール

AI が生成したテキスト成果物（プラン・説明・レビューレポート等）を、出力確定前またはスタンドアロンで再検証する正本。各コマンドは本ルールを参照し、手順を重複記載しない。

**参照（レビュー系コマンド）**: `@~/.config/shared/ai/rules/conventions/review-common-rule.md` — 調査・重要度判定・レポートテンプレはこちらに従う。本ルールは**下書きレポートの精査**のみを担う。

---

## 再検証の 3 軸（一般成果物）

| 軸           | 定義                                                                 | 検証方法の例                                                    |
| ------------ | -------------------------------------------------------------------- | --------------------------------------------------------------- |
| **誤検証**   | 根拠のない断定・存在しないファイル/API・未実行なのに成功したと言う等 | Read / Grep / 必要なら readonly な shell                        |
| **間違い**   | 事実誤認・論理矛盾・手順の誤り・数値・API の誤り                     | リポジトリ内コード・公式ドキュメント（Context7 MCP 等）との照合 |
| **ヌケモレ** | 要件の未反映・前提の未記載・代替案の欠落・MECE 違反                  | 直前 user メッセージ・`@` 添付・チケット引数との突合            |

### 成果物タイプ別（一般）

| タイプ        | 追加観点                                                              |
| ------------- | --------------------------------------------------------------------- |
| **プラン**    | Must have / Nice to have の分離、変更対象の具体性、リスク、実現可能性 |
| **説明・Q&A** | 質問への直接回答、曖昧語の排除、反例・境界条件                        |
| **混合**      | セクションごとに上記を適用                                            |

---

## レビューレポート向け再検証

親コマンドが `review-diff` / `review-pr` / `review-pr-magi` / `review-blog` のとき、またはスタンドアロンで直前応答がレビュー・評価レポート形式のときに適用する。

3 軸のマッピング:

| 軸           | レビューでの意味                                            | 優先アクション               |
| ------------ | ----------------------------------------------------------- | ---------------------------- |
| **誤検証**   | **誤指摘**（diff/コードと照合して問題ないのに NG とした等） | **削除**または重要度**降格** |
| **間違い**   | 行番号・ファイル名・因果の誤り、テンプレ不整合              | **修正**                     |
| **ヌケモレ** | 本当に修正必須なのに未記載                                  | **追加**（根拠付きのみ）     |

### 過剰レビュー抑制（レビュー専用）

- デフォルトは **削る・下げる・まとめる**。新規指摘の追加は diff/コード/記事上の根拠を示せる場合のみ
- 同趣旨の指摘は 1 件に統合する
- スタイル・好みのみの LOW は、ユーザー価値が低ければ削除または任意扱いにする
- CRITICAL/HIGH は `review-common-rule` / `pr-review-rule` の定義に照合し、満たさないものは降格する
- 集計・次のアクションを、残した指摘と整合させて更新する
- 深掘りで「念のため」列挙した候補のうち、確信が弱いものは最終レポートに載せない

**出力**: 親コマンドの規定フォーマット（`review-common-rule` のテンプレ、ブログ評価レポート形式等）を維持する。再検証用の二重レポートは出さない。

---

## サブエージェント

`review-common-rule.md` の「深掘り調査」と同型:

1. ユーザーが「サブエージェントを使って」等と明示 → `explore`（readonly）を並列起動。プラン主体の一般成果物では `design-advisor` も可
2. 明示がない → 親エージェントが直接再検証（デフォルト）
3. 判断できない → ユーザーに確認

---

## インライン再検証

当該コマンドが出力する**下書き**（まだユーザーに見せない案）を対象に再検証し、修正後の本文のみを最終出力する。

- **コンテキスト**: 当該コマンドの Input・チケット・`@` 添付・直前 user メッセージ
- **手順の選択**:
  - 親が `review-diff` / `review-pr` / `review-pr-magi` / `review-blog` → **レビューレポート向け再検証**を適用
  - それ以外 → **一般の 3 軸**を適用
- **検証サマリー**: 省略可（token 節約）。修正があった場合のみ最大 3 行（レビュー系は「削除 N 件・降格 N 件」等でよい）
- **最終版**: 当該コマンドの**規定フォーマット**に従う
- [`token-optimization-rule.md`](token-optimization-rule.md) の観点で冗長な二重出力を避ける

---

## スタンドアロン再検証

`/verify-output` から参照。会話内の**直前 assistant メッセージ**を対象とする。

- 対象がない場合: 再実行を依頼して終了
- レビューレポート形式（重要度別指摘・チェック結果サマリー、ブログ評価の 7 観点スコア等）と判断したら **レビューレポート向け再検証**を適用
- 実装完了の短文のみで diff/PR 未提示: 「`/review-diff`（staged）または `/review-pr`（PR URL）でレビューしてください」と案内して終了
- **出力**: 検証サマリー（最大 5 行）+ 最終版（直前応答と同じ体裁）
- 修正が不要: 「変更なし」と明示し、原文を最終版として再掲

---

## 共通 Guardrails

- コードファイルの書き換えは行わない
- 推測で直さない。確認不能な点は「要確認」と明記
- 検証不能な項目を断定で埋めない
- 事実確認に shell が要る場合はユーザー承認に従う
