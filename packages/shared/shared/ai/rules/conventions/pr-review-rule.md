# PR レビュールール（汎用）

重要度の分類・出力の骨格は [`review-common-rule.md`](./review-common-rule.md) に従う。本ファイルはレビュー観点と PR 出力テンプレートを定義する。

## 基本方針

### 目的

- コード品質の維持と向上
- バグの早期発見
- ドメイン知識の共有と属人性の排除
- コーディング規約とアーキテクチャルールの遵守確認

### 重要性

- **安全性**: 本番環境への不具合混入を防ぐ最後の砦です。
- **保守性**: 将来の開発者が理解しやすいコードを維持します。
- **一貫性**: プロジェクト全体での設計思想や実装パターンの統一を図ります。

---

## コンテキストの活用

### 対象リポジトリのルール

- 対象リポジトリの `.cursor/rules/`（または同等のルール配置）を動的に探索・参照する
- 共有規約（stow 後）: `@~/.config/shared/ai/rules/conventions/commit-message-rule.md`
- 共有規約（stow 後）: `@~/.config/shared/ai/rules/conventions/branch-name-rule.md`

### 詳細観点（変更内容に応じて選択適用）

- セキュリティ観点（認証・認可、入力バリデーション、機密情報の取り扱い）
- エラーハンドリング観点（例外処理、エラーメッセージ、リカバリー）
- パフォーマンス観点（N+1、不要なリクエスト、メモリリーク）

---

## レビュー観点

各変更に対して、以下の観点から確認する。プロジェクト固有のルールがある場合はそちらを優先する。

### アーキテクチャ適合性

- レイヤー境界の違反がないか（例: コントローラーから直接リポジトリを呼んでいないか）
- 依存方向が正しいか（内側のレイヤーが外側に依存していないか）
- モジュール間の依存が適切な抽象を経由しているか

### セキュリティ

- SQL/NoSQL インジェクションの可能性
- 認証/認可の漏れ（ガードの適用忘れ）
- 秘密情報（API キー、トークン）のハードコード
- 入力バリデーションの不足

### パフォーマンス

- N+1 クエリの可能性
- 不要な再レンダリング（React の依存配列、メモ化の不足）
- インデックスのないクエリ
- 大量データの一括処理（バッチ処理の検討）

### 保守性

- 命名の明確さ（曖昧な名前: `data`, `info`, `manager` を避ける）
- 単一責任の原則
- DRY（重複コードの検出）
- 既存パターンとの一貫性

### テスト

- テストが追加/更新されているか
- エッジケースがカバーされているか
- テストが実装詳細に結合していないか

## PR レビューのテンプレート

```markdown
### 判定: {✅ Approve / ❌ Request Changes / ⚠️ Comment}

{判定理由の要約。例: CRITICALおよびHIGHの指摘事項はありません。MEDIUM以下の項目について確認をお願いします。}

---

### 対象ファイル

- `path/to/file1.ts`
- `path/to/file2.ts`
- ...

### チェック結果サマリー

| カテゴリ         | 結果         |
| ---------------- | ------------ |
| アーキテクチャ   | ✅ / ⚠️ / ❌ |
| セキュリティ     | ✅ / ⚠️ / ❌ |
| パフォーマンス   | ✅ / ⚠️ / ❌ |
| 保守性           | ✅ / ⚠️ / ❌ |
| テスト           | ✅ / ⚠️ / ❌ |
| MECE（網羅性）   | ✅ / ⚠️ / ❌ |
| Over-engineering | ✅ / ⚠️ / ❌ |

---

### 🔴 CRITICAL (重大な問題)

{該当なしの場合は「_該当なし_」と記述}

- **C-1. {問題のタイトル}**
  - **箇所**: `path/to/file.ts:42`
  - **現状**: {現在のコードの問題点}
  - **理由**: {なぜこれが重大な問題なのか}
  - **推奨**: {修正案やあるべき姿}
  - **改善コード例**: {具体的なコード修正例がある場合に記載}
  - **ルール参照**: `{関連するルールファイル名}`

### 🟠 HIGH (修正必須)

{該当なしの場合は「_該当なし_」と記述}

  - **箇所**: `path/to/file.ts:42`
  - **現状**: {現在のコードの問題点}
  - **理由**: {なぜ修正が必要なのか}
  - **推奨**: {修正案やあるべき姿}
  - **改善コード例**: {具体的なコード修正例がある場合に記載}
  - **ルール参照**: `{関連するルールファイル名}`

---

### 🟡 MEDIUM (改善推奨)

{該当なしの場合は「_該当なし_」と記述。機能的な不具合ではないが、コードの健全性向上のために対応を推奨する項目}

- **M-1. {問題のタイトル}**
  - **箇所**: `path/to/file.ts:42`
  - **現状**: {現在のコードの問題点}

  - **ルール参照**: `{関連するルールファイル名}`

---

- **L-1. {確認事項のタイトル}**
  - **箇所**: `path/to/file.ts:42`
  - **現状**: {現状の記述}
  - **判断**: {変更不要と判断した理由、または軽微な修正提案}

---

### ✅ 良い点

{良い実装や工夫が見られる箇所を記載。該当なしの場合は「_該当なし_」と記述}

- **{良い点のタイトル}**
  - **箇所**: `path/to/file.ts:42`
  - **評価**: {何が良いのか、なぜ評価できるのか}

---

### 集計

- CRITICAL: X 件
- HIGH: Y 件
- MEDIUM: Z 件
- LOW: W 件

---

### 次のアクション

指摘事項を以下の 2 段階に分類し、対応方針を提示する:

- **Must have（修正必須）**: CRITICAL / HIGH の指摘。マージ前に対応が必要
- **Nice to have（改善推奨）**: MEDIUM / LOW の指摘。マージ後や別タスクでの対応可
```
