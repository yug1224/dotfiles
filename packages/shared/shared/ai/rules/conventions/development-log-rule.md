# 開発ログ記載ルール

## 基本方針

### 目的

このルールは、タスク（QA指摘、機能開発、バグ修正など）の対応完了時に、**再現性のある開発ログ**を MECE な構造で記載するためのガイドラインです。

### 重要性

1. **トレーサビリティの確保**: なぜその修正を行ったのか、後から追跡できる
2. **ナレッジの蓄積**: 同様の問題が発生した際の参考資料になる
3. **レビュアーへの情報提供**: PR内容だけでは伝わらない背景や意思決定を補完
4. **引き継ぎの円滑化**: 将来の担当者が経緯を理解しやすくなる
5. **品質保証**: 対応内容の妥当性を客観的に評価できる

---

## MECE 構造

開発ログは以下の6セクションで構成する。各セクションの責務は排他的であり、全体で対応内容を網羅する。

| セクション | 責務                          | 記載する内容                       | 記載しない内容     |
| ---------- | ----------------------------- | ---------------------------------- | ------------------ |
| 概要       | What（何を対応したか）        | タスク ID・タイトル・説明・関連 PR | 原因分析や実装詳細 |
| 分析       | Why（なぜ起きた/なぜ必要か）  | 現象・原因、または背景・目標       | 具体的な実装手段   |
| 方針       | How-Design（どう解決するか）  | 機能要件・技術要件                 | 実装の詳細手順     |
| 実装       | How-Impl（何をしたか）        | 領域別の変更内容                   | 要件の説明や背景   |
| 検証       | Verify（正しく動くか）        | 確認手順・結果                     | 実装内容の繰り返し |
| 残課題     | Follow-up（次に何をすべきか） | 後続タスク・技術的負債・既知の制限 | 今回の対応内容     |

---

## 開発ログのテンプレート

```markdown
## 概要

**[タスクID] タスクタイトル**

- 対応内容の簡潔な説明（1-2文）
- **関連PR**:
  - [#XXXX](URL)
  - [#XXXX](URL)

---

## 分析

<!-- バグ修正・QA指摘の場合 -->

**現象:**

- 具体的な問題点1
- 具体的な問題点2
- 具体的な問題点3

**原因:**

- 技術的な原因1（コード/設計レベル）
- 技術的な原因2（コード/設計レベル）

<!-- 機能開発の場合は「現象・原因」を「背景・目標」に読み替え -->
<!-- **背景:** なぜこの機能が必要か -->
<!-- **目標:** 実現したいこと -->

---

## 方針

**機能要件:**

- 実現すべき機能1
- 実現すべき機能2
- 実現すべき機能3

**技術要件:**

- 満たすべき技術的要件1
- 満たすべき技術的要件2

---

## 実装

<!-- 存在する領域のみ記載する（FE / BE / Infra / 共通 等） -->

### フロントエンド ([PR #XXXX](URL))

**1. 実装内容の大項目1**

- 具体的な実装内容（ファイル名、関数名含む）
- 技術的判断の理由

**2. 実装内容の大項目2**

- 具体的な実装内容
- 使用した技術/パターン

### バックエンド ([PR #XXXX](URL))

**1. 実装内容の大項目1**

- 具体的な実装内容
- 標準パターンへの準拠

---

## 検証

- 確認手順1 → 結果
- 確認手順2 → 結果
- CI/テストの状態

---

## 残課題

- 後続タスク: [タスクID](URL)（あれば）
- 既知の制限事項（あれば）
- 技術的負債（あれば）
- 該当なしの場合は「なし」と記載
```

---

## 記載量の目安

各セクションの推奨項目数：

| セクション       | 推奨項目数 | 各項目の長さ |
| ---------------- | ---------- | ------------ |
| 概要             | 1-2文      | 簡潔に       |
| 分析（現象）     | 3-5項目    | 1行/項目     |
| 分析（原因）     | 2-4項目    | 1-2行/項目   |
| 方針（機能要件） | 3-5項目    | 1行/項目     |
| 方針（技術要件） | 2-3項目    | 1行/項目     |
| 実装（各大項目） | 2-4個      | 2-3行/項目   |
| 検証             | 2-4項目    | 1行/項目     |
| 残課題           | 1-3項目    | 1行/項目     |

**重要**: 過度に詳細にせず、**要点を簡潔に**まとめることを優先してください。
