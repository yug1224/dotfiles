# PR概要の記述ルール

## 基本方針

### 目的

このルールは、プルリクエストの概要を標準化し、レビュアーが変更内容を迅速かつ正確に理解できるようにするためのものです。プロジェクトの PR テンプレートが存在する場合はそれを最優先し、テンプレートがない場合のみビルトインテンプレートを使用します。

### 重要性

- **レビュー効率の向上**: 明確な概要により、レビュアーは変更の文脈を素早く把握できます
- **品質保証**: チェックリストにより、重要な確認項目の漏れを防ぎます
- **ナレッジの蓄積**: 将来の開発者が変更履歴を追跡しやすくなります
- **AIとの協調**: AIが適切なPR概要を生成するための明確なガイドラインとなります

---

## テンプレートの優先順位

PR Description のテンプレートは以下の優先順位で決定する。上位が見つかった場合は、そのテンプレートの構造に従うこと。

1. **プロジェクトの PR テンプレート**（最優先）
   - `.github/PULL_REQUEST_TEMPLATE.md`
   - `.github/PULL_REQUEST_TEMPLATE/` 配下のテンプレート
   - プロジェクトの `.cursor/rules/` 配下で PR テンプレートを定義しているルール
2. **本ルールのビルトインテンプレート**（フォールバック）
   - プロジェクトにテンプレートが存在しない場合のみ使用する

**重要**: プロジェクトのテンプレートが見つかった場合は、そのセクション構成・見出し・チェックリストをそのまま使用する。テンプレートの構造を勝手に変更しない（「主な実装内容」の必須追加は除く）。

### 「主な実装内容」の必須追加（プロジェクトテンプレート利用時）

プロジェクトの PR テンプレートを尊重するが、テンプレートに **「主な実装内容」に相当するセクション**（実装の詳細・タスク単位の変更内容を列挙する見出し）がない場合は、**必ず** `### 主な実装内容` を追加する。これはテンプレート構造の逸脱ではなく、必須セクションの補完とみなす。

| 項目               | 内容                                                                 |
| ------------------ | -------------------------------------------------------------------- |
| 相当セクションの例 | `### 主な実装内容`、`## 実装詳細`、`## Changes` 内のタスク列挙など   |
| 追加しない例       | 上記に相当するセクションが既にテンプレートに含まれている場合         |
| 追加位置           | 変更内容・概要を記載するセクション（例: `## PR概要`）の直後          |
| 記載形式           | ビルトインテンプレートと同様に、番号付きリストでタスク単位に記載する |

```markdown
### 主な実装内容

1. **タスク概要**
   - タスク詳細
```

## MECE の記載原則

以下の MECE 原則は、**どのテンプレートを使う場合でも**記載内容の品質を保つための指針として適用する。テンプレートの構造自体を変えるものではない（「主な実装内容」がテンプレートに無い場合の追加は、上記の必須補完として許容する）。

| セクション       | 責務                 | 記載する内容                                               | 記載しない内容           |
| ---------------- | -------------------- | ---------------------------------------------------------- | ------------------------ |
| PR概要           | 目的・背景・変更内容 | なぜ必要か（目的1つ + 関連リンク）+ 何を変更したか（概要） | 検証手順や影響範囲の詳細 |
| 主な実装内容     | 実装の詳細           | タスク単位の変更内容（番号付きリスト）                     | 目的・背景の繰り返し     |
| 動作確認         | 検証チェックリスト   | どう検証するか（チェックリスト）                           | 変更内容の繰り返し       |
| 困ったらチェック | 影響範囲・補足情報   | どこに影響するか + レビュー時の参考情報                    | 他セクションの重複       |

## PR タイトル

PR タイトルは [コミットメッセージ規約](./commit-message-rule.md) の**ヘッダー1行**と同一形式とする。Squash merge 時に main ブランチへ入るコミットメッセージと揃えやすくするため。

```
<type>(<scope>): <subject> [<ticket-id>]
```

### ルール

- **type / scope / subject**: [コミットメッセージ規約](./commit-message-rule.md) のヘッダー定義に従う（type 8 種、scope は kebab-case で省略可、subject は日本語・末尾ピリオドなし）
- **[ticket-id]**: チケット ID が判明している場合のみ、半角スペース + 角括弧で subject の末尾に付与（例: `[DC-6713]`）
- **文字数**: ヘッダー全体 72 文字以内（subject 50 文字以内、`[TICKET-ID]` 含む）
- **複数コミット PR**: 主目的 1 件に合わせた 1 タイトルとする
- **優先順位**: プロジェクトに PR タイトルの慣習やテンプレートで別形式が定められている場合はそちらを優先する（Description のテンプレート優先と同様）

### 例

```
feat(user-list): フィルタリング機能を追加 [DC-6713]
fix(api-client): null応答のハンドリングを修正
chore: 依存関係を更新 [PROJ-99]
fix(csv-import): 最終更新日をインポートで編集不可にする [DC-6713]
```

### ✅ Good / ❌ Bad

| 区分 | 例                                                                        |
| ---- | ------------------------------------------------------------------------- |
| Good | `feat(user-list): フィルタリング機能を追加 [DC-6713]`                     |
| Good | `fix(api-client): null応答のハンドリングを修正`                           |
| Bad  | `最終更新日をCSVインポートで編集不可にする [DC-6713]` → type / scope なし |
| Bad  | `feat: 一覧とユーザー管理を追加` → 複数変更を1タイトルに含めている        |

## 基本ルール

- PR タイトルは上記「PR タイトル」に従うこと
- PR概要は、必ず日本語で記述すること
- テンプレートのすべてのセクションを埋めること
- 「目的」は該当するものを **1つだけ選んで** 記載すること（列挙しない）
- プレースホルダー（`todo`, `タスク概要`）は必ず具体的な内容に置き換えること
- 具体的で理解しやすい表現を使用すること
- **プロジェクトの PR テンプレートが存在する場合は、そのテンプレートの構造から逸脱しないこと**（ただし「主な実装内容」に相当するセクションがない場合は、上記「必須追加」に従い `### 主な実装内容` を追加すること）

## ビルトインテンプレート（フォールバック用）

以下のテンプレートは、プロジェクトに PR テンプレートが存在しない場合のみ使用する。

### セクションごとの変更ルール

| セクション                          | 変更可否                 | 説明                                     |
| ----------------------------------- | ------------------------ | ---------------------------------------- |
| `## 📝 PR概要 📝`                   | 内容変更を推奨           | プレースホルダーを適切な内容に置き換える |
| `### 主な実装内容`                  | 内容変更を推奨           | プレースホルダーを適切な内容に置き換える |
| `### 変更前後のスクリーンショット`  | 内容変更を推奨           | スクリーンショットを貼付する             |
| `## 👮‍♂️ 動作確認 👮‍♂️`                 | リストの追加・変更は不可 | チェックを入れることのみ可               |
| `## ✅ 困ったらこの辺をチェック ✅` | 変更不可                 | テンプレートの内容をそのまま維持する     |

### バックエンドの PR 概要テンプレート

バックエンド系リポジトリ、またはフロントエンド以外のリポジトリで使用するテンプレート。

```markdown
## 📝 PR概要 📝

- **目的**:
  - 機能追加
  - 仕様変更
  - バグ修正
  - リファクタリング
- **関連リンク**:
  - [todo](todo)
- **変更点の概要**:
  - todo

### 主な実装内容

1. **タスク概要**
   - タスク詳細

## 👮‍♂️ 動作確認 👮‍♂️

- [ ] ローカルで動作確認済み
- [ ] CI が正常に通過

## ✅ 困ったらこの辺をチェック ✅

- 影響するエンドポイント / モジュール
- 破壊的変更の有無
- 補足情報があれば記載
```
