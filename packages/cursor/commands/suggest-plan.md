name: suggest-plan
description: 要件から設計・実装計画を Advisor で策定し松竹梅で提案する
---

要件（Notion チケット / GitHub Issue / 自然言語）を起点に、`design-advisor` と `build-advisor` で設計・実装の両面から分析し、松竹梅の実装計画を策定する。全ステップが読み取り専用のため、Plan モードでの実行を推奨する。

**参照ルール**: `~/.cursor/rules/checklists/nfr-checklist.mdc`

**Input**: `/suggest-plan` の後に続く引数は、要件の説明文、チケット ID またはチケット URL（任意）。Notion、GitHub Issues 等のチケットシステムに対応。

**使用例**:

- `/suggest-plan ユーザー一覧画面にフィルタリング機能を追加`
- `/suggest-plan https://notion.so/xxx`
- `/suggest-plan https://github.com/owner/repo/issues/123`

---

## Steps

### 0. トレース（必須）

応答の冒頭に `Applied: /suggest-plan` と出力する。

### 1. 要件の収集

以下の情報源を**並列で**収集する。

a. **チケット情報**（引数がある場合）

引数にチケット ID または URL が渡された場合、適切な手段でチケットの目的・背景・タイトル・受け入れ条件を取得する。

チケットシステムの判別と取得方法:

- **Notion**: Notion MCP を使用（URL に `notion.so` / `notion.site` を含む場合、または Notion ページ ID 形式の場合）
- **GitHub Issues / PR**: GitHub MCP または URL フェッチを使用（URL に `github.com` を含む場合）
- **その他の URL**: URL フェッチで内容を取得する
- **ID のみ（URL なし）**: ユーザーに補足情報を確認するか、会話コンテキストから推測する

b. **コンテキスト情報**

ユーザーが `@` で添付したファイルや会話中で提供した情報があれば、それも考慮する。

c. **プロジェクト構造**

ワークスペースルートで以下を実行し、プロジェクトの技術スタックとディレクトリ構成を把握する。`.cursor/rules/` 内の関連ルールがあれば Read で確認する。

```bash
ls -la
```

### 2. 要件の整理と確認

収集した情報を以下の形式で整理し、ユーザーに共有する:

- **目的**: 何を実現するか（1-2文）
- **スコープ**: 変更が必要な領域（FE / BE / DB / インフラ等）
- **制約**: 技術的制約・期限・互換性要件等

方向性に認識齟齬がないか確認してから、次のステップに進む。

### 3. Advisor による設計・実装分析

Task ツールで以下の 2 つの Subagent を**1つのメッセージ内で同時に**起動する（逐次起動しない）。

**a. 設計分析（`design-advisor`, `readonly: true`）**

要件とプロジェクト構造を渡し、以下を分析する:

- アーキテクチャ上の影響（レイヤー構成、モジュール間依存）
- API 設計（新規エンドポイント / 既存変更の必要性）
- DB スキーマ変更の必要性（マイグレーション要否）
- 非機能要件への影響（`nfr-checklist.mdc` の観点）

**b. 実装分析（`build-advisor`, `readonly: true`）**

要件とプロジェクト構造を渡し、以下を分析する:

- 既存の実装パターンとの整合性
- 変更対象ファイルの特定と変更内容の概算
- テスト戦略（追加・更新が必要なテスト）
- 実装上のリスクと注意点

### 4. 松竹梅の実装計画を策定

Advisor の分析結果を統合し、以下の 3 段階で計画を策定する。

- **松（理想）**: 設計品質・テスト・非機能要件を十分に考慮した包括的な実装
- **竹（推奨）**: 実用的なバランスで、主要な品質を担保しつつ効率的に実装
- **梅（最小）**: 要件を満たす最小限の変更（技術的負債を許容）

### 5. Markdown 形式で出力

下記フォーマットに従って出力する。

---

## 出力フォーマット

```markdown
## 実装計画

### 要件サマリー

（目的・スコープ・制約の要約）

### 設計概要

（mermaid 図を含むアーキテクチャ・データフローの概要）

### 変更対象

| ファイル / モジュール | 変更種別           | 概要           |
| --------------------- | ------------------ | -------------- |
| `path/to/file.ts`     | 新規 / 修正 / 削除 | 変更内容の概要 |

### 松（理想）

- 実装内容の詳細
- 追加テスト
- 想定工数

### 竹（推奨）

- 実装内容の詳細
- 追加テスト
- 想定工数

### 梅（最小）

- 実装内容の詳細
- 想定工数
- 許容する技術的負債

### リスクと注意点

- リスク1
- リスク2
```

## Guardrails

- `/suggest-plan` は調査と計画策定のみを行い、コードの変更は行わない
- Advisor は `readonly: true` で安全に分析する
- 要件が曖昧な場合は Step 2 でユーザーに確認してから先に進む
- 出力には必ず mermaid 図を含め、設計の全体像を視覚化する
- 各松竹梅プランには変更対象ファイルと想定工数を明記する
- 既存の実装パターン・ルールとの一貫性を最優先にする
- チケット情報の取得は引数で ID/URL が渡された場合のみ実行し、渡されなければスキップする
